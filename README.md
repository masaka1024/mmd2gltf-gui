MMDのPMXモデル(+VMDモーション)を、可能な限り忠実に **glTF 2.0(.glb)** へ変換するツールです。**GUIとCLIの両方**で使えます。依存は標準ライブラリのみで動作し、テクスチャ変換(BMP/TGA/sph/spa→PNG)にのみPillowを使います。

髪・スカートなどの**物理演算をボーンアニメーションとして焼き込む機能**を搭載。物理エンジンを持たないglTFビューアでも、髪やスカートが自然に揺れた状態で再生できます。

このツールの本来の狙いは、単なるビューア向け変換ではなく、**MMDのデータを他のフォーマット・他の環境へ可能な限り正確に持ち出すこと**です(詳しくは[このツールの狙い](#このツールの狙い)を参照)。

> **▶ すぐ使う(Windows):** [最新版のEXEをダウンロード](https://github.com/masaka1024/mmd2gltf-gui/releases/latest) — Pythonのインストール不要。

## このツールの狙い

mmd2gltfの本来の目的は、**MMDのデータを他のフォーマット・他の環境へ、可能な限り正確に持ち出すこと**です。glTF(.glb)は、そのための「運び出す器(コンテナ)」として使っています。

このため、変換結果は次の**二層構造**になっています。

1. **保存層(`extras.mmd`)** — 剛体・ジョイント・IK設定・付与親・材質のトゥーン/スフィア/エッジ設定など、glTFでは表現できない情報も**生のPMX値のまま完全保存**します。受け取り側(ゲームエンジンや他ツール)は、この層から元のMMDモデルの構造を高い精度で再構築できます。
2. **ベイク層(glTF標準のアニメーション)** — 物理エンジンを持たない一般のglTFビューアでも「そのまま自然に見える」ように、モーションと揺れ物を標準キーフレームに焼き込みます。こちらは近似的なフォールバックという位置づけです。

物理ベイクを使っても保存層は失われないため、「ビューアではベイク済みの揺れをそのまま再生し、エンジンでは生データから本物の物理を再構築する」という使い分けができます。

### 実証: Unityインポーター

この設計が実際に機能することを確かめるため、`extras.mmd` を読み取ってUnity上でモデルを再構築するエディタ拡張を開発しました。以下の動作を複数モデルで確認しています。

- 剛体・ジョイント情報からの**物理(揺れ物)の再構築** — Rigidbody / ConfigurableJoint をボーンに構成し、髪・スカートがUnityの物理エンジンで揺れる状態を復元
- 材質のトゥーン/スフィアマップ/共有トゥーン/輪郭線設定からの**MMD風トゥーン表示の復元**(lilToonベース)

つまり本ツールで変換した .glb は、「ビューアでそのまま見られる」だけでなく、**エンジン側で本物の物理とトゥーン表現を取り戻せるだけの情報**を保持しています。

> インポーター本体は別リポジトリとして公開しています: **[mmd2gltf-unity-physics-importer](https://github.com/masaka1024/mmd2gltf-unity-physics-importer)**(要 UniGLTF / lilToon、URPプロジェクト向け)

## 特徴

- **追加ライブラリ不要** — 標準ライブラリのみで動作(テクスチャ変換時のみ任意でPillow)。
- **PMX 2.0/2.1 を全セクション解析** — メッシュ・スキニング・材質・モーフ・物理設定まで読み込み。
- **VMDモーションをベイク** — ベジェ補間を評価し、MMDと同じ変形順(変形階層→付与→CCD-IK+軸制限)で毎フレーム焼き込み。
- **物理演算のベイク** — 髪・ネクタイ・スカートなどの剛体物理をシミュレートし、ボーンキーフレームとして焼き込み。物理エンジンのないビューアでも揺れ物が動きます。
- **IKの扱いを4通りで細かく制御** — フルベイクから「一切解かない」まで([IKの扱い](#ikの扱い)を参照)。
- **glTFで表現できない情報は失わない** — 剛体・ジョイント・IK設定・付与親などを `extras.mmd` に生データで完全保存。物理ベイクを使っても元の物理設定はそのまま残るため、Unity等のエンジン側で本物の物理を再構築することも可能です([Unityインポーター](https://github.com/masaka1024/mmd2gltf-unity-physics-importer)を公開中)。
- **GUI / CLI 両対応** — GUIは日本語/English切り替え、ドラッグ&ドロップ対応。
- **Windows EXE版あり** — Pythonを入れなくても、ダウンロードしてすぐ使えます。

## IKの扱い

VMDモーションを焼き込むとき、IK(足・腕などの逆運動学)をどう処理するかを4通りで選べます。**通常は既定のフルベイクのままでOK**で、意図しない足の動きが出たときに切り替えます。

| モード | 指定 | 挙動 |
| --- | --- | --- |
| **フルベイク(既定)** | `--vmd FILE` のみ | 全IKを解決してアニメーションに焼き込む(30fps)。VMD内のIK ON/OFFキーは尊重する。 |
| **全IKを解かない** | `--no-ik` | ベイク時にIKを一切解かず、生のFKカーブのみを出力する。 |
| **一部だけ無効化** | `--disable-ik 名前` | 名前を含むIKボーンだけ無効化(繰り返し可)。例: `--disable-ik 足` で足/つま先IKだけ切る。 |
| **VMDのIKキーを無視** | `--ignore-vmd-ik` | VMD内のIK ON/OFFキーを無視して解く。フルキー版モーション等で足IKがOFF指定されていると既定では自動で解かないため、それを打ち消したいとき。 |

**選び方の目安**

- ふつうのダンスモーション → 既定(`--vmd` のみ)のままで問題ありません。
- 生のカーブを確認したい・IK解決の結果が気に入らない → `--no-ik`。
- 足だけ/腕だけなど特定のIKを切りたい → `--disable-ik 足` のように部分指定。
- フルキー版などでVMD側のIK OFF指定によって足が正しく追従しない → `--ignore-vmd-ik`。

## 物理ベイク

`--bake-physics` を付けると、MMDの剛体物理(髪・ネクタイ・スカートなどの揺れ物)を簡易物理シミュレーション(PBDばねモデル)で解き、結果を**ボーンのキーフレームとして焼き込み**ます。物理エンジンを持たないglTFビューアでも、揺れ物が自然に動いた状態で再生されます。

> **設計方針:** このベイクは「物理エンジンがない環境向けの、今風でもっともらしい揺れ」を目指した近似です。MMD本体(Bullet物理)の完全再現は目標としていません。忠実な物理が必要な場合は、`extras.mmd` に保存された生の剛体・ジョイント情報を使い、Unity等のエンジン側で再構築してください。

### 基本の使い方

```bash
# 髪だけベイク(既定)
python -m mmd2gltf モデル.pmx --vmd モーション.vmd --bake-physics -o out.glb

# スカートも含めて全部ベイク
python -m mmd2gltf モデル.pmx --vmd モーション.vmd --bake-physics --bake-target all -o out.glb
```

- `--bake-target hair`(既定) — 髪・ネクタイなどチェーン状の揺れ物のみ。
- `--bake-target all` — スカートを含む全ての動的剛体。スカートはリング構造を保つ布ソルバーで処理し、体のコライダー(足・腰など)との衝突判定・貫通防止も行います。

### 揺れの調整

| オプション | 既定 | 効果 |
| --- | --- | --- |
| `--hair-drag F` | 0.85 | 速度の保持率(0〜1)。上げるとふわふわ、下げると硬めに。 |
| `--hair-stiffness F` | 1.5 | 元の形に戻ろうとする力。 |
| `--hair-gravity F` | 0.02 | 重力の強さ。0で初期形状を維持。 |
| `--collision-margin F` | 0.01 | 布と体コライダーの間に確保するクリアランス(glTF単位)。スカートが脚に触れて見えるときに増やす。 |
| `--hem-extra-margin F` | 0.0 | スカートの**裾側だけ**に追加するクリアランス。根元(腰側)から裾へ滑らかに補間されるため、腰まわりのシルエットを変えずに裾の食い込みだけを抑えられます。 |

### 衝突モード(食い込み・傘化への対処)

一部のモデルでは、体内部の剛体設定やモデルの想定外の激しいモーションが原因で、スカートが押し広げられて傘のように開いたり、特定の部位に食い込んだりすることがあります。そのための逃げ道を2方式用意しています。

| モード | 指定 | 挙動 |
| --- | --- | --- |
| **通常(デナイリスト)** | `--force-no-collision 剛体名`(繰り返し可) | 指定した剛体だけを完全非衝突にする。それ以外はPMXの非衝突グループ設定どおり。「この剛体が悪さをしている」と分かっているときに。 |
| **限定(アローリスト)** | `--allowed-collider 剛体名`(繰り返し可) | 指定した剛体**だけ**をコライダーとして扱い、それ以外は最初から存在しないものとする(PMXのグループ設定は無視)。VRM SpringBone / VRChat PhysBonesと同じ考え方。 |

GUIでは「詳細設定」から同じことがもっと簡単にできます:

- **衝突モード** — ラジオボタンで「通常/限定」を切り替え、剛体名をカンマ区切りで入力。
- **剛体名を確認...** — 選択中のPMXを解析し、剛体を「静的(コライダー)/揺れ物」に分けて一覧表示。ダブルクリックで入力欄に追加できます。
- **VRM互換モードにする** — 脚まわりの剛体を自動抽出し、ワンクリックで「限定」モードに設定するプリセット。

PMXの非衝突グループ(group/noCollisionMask)設定自体も正しく読み込んで反映します。上記のモードは、その設定では対処しきれない場合の上書き手段です。

## オプション一覧

| オプション | 説明 |
| --- | --- |
| `--vmd FILE` | VMDモーションをglTFアニメーションとしてベイク(IK解決込み・30fps) |
| `--no-ik` | ベイク時に全IKを解かない(生のFKカーブのみ) |
| `--disable-ik 名前` | 名前を含むIKボーンだけ無効化(繰り返し可。例: `--disable-ik 足` で足/つま先IK) |
| `--ignore-vmd-ik` | VMD内のIK ON/OFFキーを無視する(既定では尊重。フルキー版モーション等で足IKがOFF指定されていれば自動で解かない) |
| `--bake-physics` | 剛体物理をボーンキーフレームに焼き込む(PBDばねシミュレーション)。IKで固まった髪・ネクタイ等が動くようになる |
| `--bake-target MODE` | ベイク対象。`hair`=髪のみ(既定)、`all`=スカート含む全動的剛体(布ソルバーがリング構造に対応) |
| `--hair-drag F` | 速度保持率0〜1(既定0.85。上げるとふわふわに) |
| `--hair-stiffness F` | 初期形状への復元力(既定1.5) |
| `--hair-gravity F` | 重力の強さ(既定0.02。0で初期形状維持) |
| `--collision-margin F` | 布と体コライダー間のクリアランス、glTF単位(既定0.01)。スカートが脚に触れて/めり込んで見えるときに増やす。0で表面ぴったりまで押し出し |
| `--force-no-collision 名前` | 指定した剛体を強制的に完全非衝突にする(繰り返し可)。PMXのグループ設定が正しく見えるのに極端なポーズで布が引っかかる/伸びる場合の逃げ道 |
| `--allowed-collider 名前` | アローリストモードに切替: 指定した剛体だけをコライダーとして扱う(繰り返し可。1つでも指定するとPMXのgroup/noCollisionMask判定を完全にバイパス)。VRM SpringBone / VRChat PhysBones流の設計 |
| `--hem-extra-margin F` | `--collision-margin` に上乗せする裾専用クリアランス。各スカートチェーンの根元0→裾1で滑らかに補間して適用されるため、腰側のシルエットに影響しない(既定0.0=無効) |
| `--step N` | Nフレームおきにサンプリングして容量削減(既定1=全フレーム) |
| `--unlit` | 材質に `KHR_materials_unlit` を付与(MMDのトゥーン見た目に近い) |
| `--no-extras` | `extras.mmd`(下記)を出力しない |
| `--anim-name NAME` | アニメーション名 |
| `--morph-mode MODE` | モーフの格納方式。`sparse`=軽量(既定)、`dense`=最大互換(sparse非対応ビュアーで顔が壊れる場合はこれ)、`none`=モーフ無し |
| `--alpha-mode MODE` | `auto`(既定)はテクスチャのα分布を解析し OPAQUE/MASK/BLEND を自動判別。肌テクスチャの未使用領域の透明を誤ってBLENDにしない(顔が透ける・裏返って見える問題の対策)。`opaque`/`mask`/`blend`で全材質強制も可 |
| `--force-double-sided` | 全材質を両面描画に(three.js MMDLoaderと同じ挙動。髪やスカートの裏面が消える場合に) |
| `--scale F` | MMD単位→glTF単位(メートル)への一律スケール(既定0.08)。MMDモデルは慣習的に1ユニット≈8cmで作られており(身長160cmなら約20ユニット)、`1.0`のまま(無変換)にするとglTFビューアでは約12.5倍の巨人として表示される。既定の`0.08`はそのための変換係数。頂点・ボーン位置・SDEFパラメータ・モーフの位置オフセット・ベイク済みアニメーションの平行移動に適用され、法線・UV・回転・`extras.mmd`内の生データ(剛体・ジョイント等)には影響しない(`extras.mmd.unitScale`に適用した係数を記録)。元モデルが別の単位系で作られている場合は`--scale 1.0`などに調整 |
| `--no-custom-attrs` | MMD固有の頂点属性(`_SDEF_C`/`_SDEF_R0`/`_SDEF_R1`/`_ADDUV1..4`/`_EDGESCALE`/`_WEIGHTTYPE`)を出力しない。**Blenderの標準glTFインポータがこれらの属性名でメッシュを読み込めずエラーになる場合はこれを指定**(データはそのまま`extras.mmd`側には残るので情報は失われない) |

## ビュアー互換性の注意

- macOSのQuick Look/プレビュー(RealityKit)はglTFのモーフターゲット自体に非対応です。モーフが「抜ける」のはビュアー側の制限で、ファイルは正常です。
- sparseアクセッサの実装が不完全なビュアー/ローダーではモーフ適用時にメッシュが壊れる(顔が透ける・口パクが別の場所の変形に見える等)ことがあります。その場合は `--morph-mode dense` で変換してください(容量は増加)。**UniGLTF/UniVRM系のUnityインポータで使う場合も `dense` を推奨**します。sparse時もゼロ基底bufferViewを持たせているため、sparse未対応ローダーでは「モーフ無効」に安全に落ちます。
- 動作確認済みの推奨ビュアー: three.js系 (gltf-viewer.donmccurdy.com)、Babylon.js Sandbox (sandbox.babylonjs.com)、Blender 3.x以降のglTFインポータ。

## 変換内容(忠実度)

glTFで直接表現できるもの:

- メッシュ(頂点・法線・UV・材質ごとのプリミティブ分割)
- スキニング(BDEF1/2/4。SDEF/QDEFは線形ブレンド近似+元パラメータを保持)
- ボーン階層(PMXボーン順=skin.joints順なのでインデックス互換)
- 頂点モーフ→モーフターゲット(sparseアクセッサで軽量)、UVモーフ→`TEXCOORD_0`ターゲット、グループモーフ→展開して合成ターゲット
- 材質(拡散色→baseColor、両面フラグ、αによるBLEND判定、テクスチャ埋め込み)
- VMDモーション: ベジェ補間を評価し、MMDと同じ変形順(変形階層→付与→CCD-IK+軸制限)で毎フレームベイク。モーフキーはweightsアニメーションに変換
- 物理(任意・`--bake-physics`): 剛体物理をシミュレートしてボーンの回転+平行移動キーフレームとして焼き込み。PMXの非衝突グループ(group/noCollisionMask)を尊重

glTFに存在しない概念は `extras.mmd`(生のPMX値・MMD左手系座標)として全て保存:

- 剛体・ジョイント(物理)、IK設定、付与親、固定軸/ローカル軸、表示枠
- ボーンモーフ・材質モーフ・フリップ/インパルスモーフの内容
- 材質のスフィアマップ/トゥーン/エッジ/環境光/反射光設定、メモ
- 頂点単位データはカスタム属性として保持: `_ADDUV1..4`、`_EDGESCALE`、`_SDEF_C/_SDEF_R0/_SDEF_R1`、`_WEIGHTTYPE`

座標変換: 位置/法線 `(x,y,z)→(x,y,-z)`、クォータニオン `(x,y,z,w)→(-x,-y,z,w)`、三角形は巻き順反転。

## 制限

- PMD(旧形式)・PMX2.1のソフトボディは未対応(PMXへの変換はPMXEditor等で)
- 物理ベイクは簡易シミュレーションによる近似で、MMD本体(Bullet)の挙動を完全再現するものではありません。衝突判定はボーン(剛体)単位で、メッシュ頂点単位ではないため、素早い動きでは布のメッシュ面が体を視覚的に横切ることがあります。また、モデルの剛体配置が想定していない極端なモーションでは破綻することがあります(これはMMD本体でも起こりえます)。忠実な物理が必要な場合は `extras.mmd` の生データからエンジン側で再構築してください
- MMDのトゥーンシェーディング/スフィアマップ/エッジ描画はglTFのPBRでは再現不可のため、見た目はビューア依存です(`--unlit`で近づきます)
- 共有トゥーン(toon01〜10.bmp)はMMD本体同梱のため埋め込まれません(番号はextrasに保持)
- VMDのカメラ・照明・セルフ影キーは対象外
